# zensim-validate refactoring plan

## What this crate does

Validates zensim's metric quality by correlating its scores against human
subjective ratings (from academic IQA datasets) and against reference metrics
(from synthetic training data). Also trains the 156-element weight vectors
that ship in `zensim::profile`.

Currently a single 2700-line `main.rs`.

## External dependencies

### coefficient (data generation)

The synthetic training data is generated by a separate crate:
`/home/lilith/work/coefficient/examples/generate_zensim_training.rs`

That tool:
1. Takes source images and encodes them at various codecs/quality levels
2. Decodes back to PNG
3. Runs reference metrics (ssimulacra2, butteraugli) on each source/decoded pair
4. Outputs `training.csv` with columns: source_path, decoded_path, codec,
   quality, width, height, gpu_ssimulacra2, gpu_butteraugli, etc.
5. Outputs `metric-ledger.jsonl` tracking what was generated

zensim-validate then consumes that CSV: it computes zensim features for each
pair and trains weights to maximize SROCC against the reference metric column.

### Academic IQA datasets

zensim-validate loads 7 dataset formats, each with its own directory layout
and score normalization:

| Format | Source | Score type | Normalization |
|--------|--------|------------|---------------|
| TID2013 | `mos_with_names.txt` | MOS 0-9 | /9 |
| KADID-10k | `dmos.csv` | DMOS 1-5 | (v-1)/4 |
| CSIQ | `csiq.DMOS.xlsx` or CSV | DMOS 0-1 | 1-v |
| PIPAL | `Train_Label/*.txt` | Elo ~900-1850 | min-max |
| CID22 | `CID22_validation_set.csv` | MCOS 0-100 | /100 |
| KonFiG-IQA | `DATA/EXP_III/data3.csv` | DCR 0-4 | (4-v)/4 |
| Synthetic | `training.csv` from coefficient | ssim2 or butteraugli | custom |

CSIQ requires the `calamine` crate for xlsx parsing.

### zensim training API

Uses training-gated functions from zensim:
- `compute_zensim_with_ref_and_config` — feature extraction with custom config
- `precompute_reference_with_scales` — precompute reference for batch comparison
- `score_from_features` — apply weights to feature vector
- `WEIGHTS`, `FEATURES_PER_SCALE` — embedded constants

## Proposed module structure

### `datasets.rs` (~650 lines)
All 7 dataset loaders + shared types.
- `ImagePair`, `DatasetFormat`, `TargetMetric`
- `load_tid2013`, `load_kadid10k`, `load_csiq`, `load_pipal`,
  `load_cid22`, `load_konfig_iqa`, `load_synthetic`
- `load_pairs` (dispatch by format)

### `cache.rs` (~200 lines)
Binary feature cache (avoids recomputing 149k feature vectors).
- `CacheConfig`, `CachedFeatures`
- `save_feature_cache`, `load_feature_cache`
- Format: magic `ZSFC`, version 2, config validation, row-major f64 features

### `training.rs` (~450 lines)
Weight optimization. Currently duplicated between single-dataset and
multi-dataset variants — should be unified.
- Coordinate descent with Pearson pre-filter + Spearman confirmation
- 10-20 random restarts with diverse initialization strategies
- Weight normalization (target median distance ~1.7)
- `fast_srocc` (inline Spearman with reusable buffers)
- LCG PRNG for deterministic restarts

The single-dataset path (`train_weights`) uses O(1) Pearson as a pre-filter
before computing Spearman on top candidates. The multi-dataset path
(`train_weights_multi`) skips Pearson and evaluates a blended objective
(0.5 * mean SROCC + 0.5 * min SROCC) directly. These could share the
core coordinate descent loop with a trait or strategy parameter.

### `stats.rs` (~80 lines)
Correlation statistics, no external deps.
- `spearman_correlation`, `pearson_correlation`, `kendall_correlation`, `ranks`

### `cv.rs` (~180 lines)
Cross-validation modes.
- `run_kfold_cv` — K-fold split by reference image
- `run_leave_one_out` — leave-one-dataset-out
- `make_folds` — deterministic Fisher-Yates shuffle + round-robin

### `main.rs` (~400 lines)
CLI args, orchestration, reporting.
- `Args` struct (clap)
- Feature computation (`load_and_compute`)
- Reporting (`report_embedded_correlations`, `print_trained_results`)
- Training log and weights file I/O

## Other cleanup

- **Image loading**: uses `image::open().to_rgb8()` which silently discards
  alpha, strips 16-bit, ignores color profiles. Fine for the academic datasets
  (all 8-bit RGB) but worth documenting.
- **Stale TargetMetric variants**: `CpuSsim2` and `CpuButteraugli` exist for
  the v2 CSV which stored both CPU and GPU versions. Future CSVs should only
  have one version per metric. Consider simplifying to just `Ssim2` and
  `Butteraugli` with column name fallback.
- **`expand_embedded_weights`**: only needed when num_scales != 4. Could be
  simplified or removed if we never ship profiles with different scale counts.
